Correction after repository review:

- Removed the Promise/`.execute()` wrapper approach from repositories.
- Added TypeScript module augmentation in `packages/db/src/drizzle.ts` so Drizzle query builders are typed as yieldable `Effect`s, matching the runtime Effectable patch and the `database-native` reference pattern.
- Repository implementations now use `yield* db.insert/select/update/delete...` directly inside `Effect.gen`.
- Repository method effects intentionally retain `SqlClient.SqlClient` in their requirements, so callers can compose them under `SqlClient.withTransaction(...)` and have all Drizzle calls participate in the transaction through the current fiber context.
- Replaced the deleted wrapper helper with a tiny `RepositoryUtils.first` helper only.
- Validation remains green: `bun run check:types` and `bun run test` pass.

Repository review follow-up:

- Removed `RepositoryError` entirely rather than wrapping SQL failures.
- Repository methods now expose `SqlError` directly.
- Added `Effect.fn("Repository.method")` names to repository methods for traceable operation boundaries instead of storing operation names in an error wrapper.
- Adjusted Drizzle query typing/runtime so Drizzle query Effects no longer leak `SqlClient.SqlClient` in their requirements. Runtime uses the current fiber SQL client when one is available (so transactions can still work), otherwise falls back to the construction context from `Database.layer`.
- Repository APIs now return effects requiring no external `SqlClient`; `SqlClient` is provided by the database/default layers.
- Validation: `bun run check:types` and `bun run test` pass.
